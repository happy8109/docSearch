# 技术规范：异构系统远程 SQL 执行 (RSE) 协议

## 1. 概述
本规范定义了一套用于异构系统间进行远程数据库查询的标准协议。旨在通过通用的 API 封装，将传统的“本地数据库直接访问”模式转化为“远程指令透传”模式，以实现跨网络环境的数据交互，同时最大限度减少对既有业务逻辑的修改。

## 2. 服务端实现标准 (数据提供方)
服务端需提供一个统一的入口，通过标准接口暴露受限的数据库查询能力。

### 2.1 接口契约
- **传输协议**: HTTP/HTTPS
- **请求方法**: `POST`
- **数据格式**: `application/json`
- **请求参数 (JSON)**:
  - `sql` (string): 待执行的标准 SQL 语句。
  - `params` (array): SQL 预处理参数列表（防止 SQL 注入）。

### 2.2 操作准则
1. **全指令支持**：网桥应支持标准 SQL 的全集操作，包括但不限于 `SELECT`, `INSERT`, `UPDATE`, `DELETE` 等。这使得远程系统不仅可以进行数据查询，还可以进行数据同步和状态更新。
2. **错误处理**：若 SQL 执行失败，应返回包含详细数据库引擎错误信息的 JSON。

### 2.3 响应格式
无论执行成功与否，均需返回统一的 JSON 结构：
```json
{
  "success": true,        // 执行状态
  "data": [ ... ],        // 结果集数组（执行成功时）
  "error": "Error Message" // 错误描述（执行失败时）
}
```

## 3. 客户端重构方法 (数据调用方)
客户端改造应遵循“接口一致性”原则，确保业务逻辑层无感知。

### 3.1 适配器模式 (Adapter Pattern)
开发人员应封装一套模拟数据库驱动接口的对象（抽象层）。该对象的方法签名均需与原系统使用的数据库连接库（如 sqlite3, mysql2 等）保持高度一致。
- **内部逻辑**：将本地 SQL 执行动作封装为 HTTP 请求，发送至服务端接口。
- **异步处理**：所有调用必须支持 Promise 或 Callback 机制，以应对网络传输产生的物理延迟。

### 3.2 模式切换
建议通过全局变量或配置项，实现“本地数据库驱动”与“远程 RSE 驱动”的无缝切换，确保在开发环境与生产部署环境间的快速移植。

## 4. 低带宽环境性能指标
由于底层传输链路可能存在带宽限制，应用层必须强制执行以下标准：

1. **按需取字段**：严禁使用 `SELECT *`。查询必须明确列出所需字段名，以减少无效数据传输。
2. **强制分页**：所有查询必须显式包含 `LIMIT` 和 `OFFSET`（或等效语法）。
3. **结果集控制**：建议单次请求返回的结果集 JSON 大小不超过 50KB。
4. **延迟优化**：前端 UI 必须实现非阻塞加载状态（Loading），并具备合理的超时处理机制（建议超时阈值为 15-30 秒）。
