# 局域网文档搜索引擎总体技术方案

该方案旨在构建一个轻量级、高性能的局域网文档搜索引擎，支持对 `.doc` 和 `.docx` 文件的文本提取和全文检索。前端界面将参考 Google 搜索引擎的设计，提供简洁易用的搜索体验。

## 架构概览

由于总容量较小（约6000个文件，200MB），不需要引入 Elasticsearch 等重型组件。系统采用前后端分离架构，均在一个轻量级 Node.js 进程中运行：

### 1. 核心技术栈
- **后端服务**: Node.js + Express (轻量易用，快速搭建 API 和静态文件服务)
- **数据库引擎**: SQLite3 (使用其内置的 FTS5 扩展实现极速文本全文检索)
- **中文分词方案**: `nodejieba` (结巴分词的 C++ 绑定 Node 版本，性能极高)
- **文档解析库**: 
  - `.docx`：`mammoth` 或 `textract` 等。
  - `.doc`：基于 Node.js 的 `word-extractor`，或者通过 `textract` 自动调用系统中的命令行工具（例如 `antiword`）。
- **前端页面**: 纯原生 HTML/CSS/Vanilla JS 编写静态文件（或极简 Vite 环境），CSS 模仿 Google 的极简视觉风格，保持轻量级，便于日后维护。

### 2. 项目目录结构与模块解耦
为了保证代码的清晰性和可维护性，后端代码将划分为以下高内聚、低耦合的模块：
```
doc-search-engine/
├── config/
│   ├── app.config.json    # 采用标准 JSON 格式的统一独立配置文件
│   └── index.js           # 解析并加载 app.config.json 返回给系统
├── src/
│   ├── app.js             # Express 应用实例配置与中间件注册
│   ├── server.js          # 服务启动入口
│   ├── routes/            # API 路由定义 (解耦网络层)
│   ├── controllers/       # 请求处理逻辑 (校验参数、调用 Service 层并返回 HTTP 响应)
│   ├── services/          # 核心业务逻辑层 (如搜索逻辑、同步逻辑，高度独立)
│   ├── models/            # 数据库访问层 (封装 SQLite 查询)
│   ├── utils/             # 通用工具 (如文件系统处理、日志等)
│   └── parser/            # 独立的文档解析模块 (专门负责各种格式的文本文档提取)
├── public/                # 纯原生前端静态文件 (HTML/CSS/JS)
├── db/                    # SQLite 数据库文件存放目录
├── docs/                  # 系统开发与接口 API 文档存放处
└── package.json
```
- **配置中心化**: 所有系统关键设置（如监听端口默认 3004、被扫描的文档目录、数据库路径等）都在 `app.config.json` 文件中集中化配置，系统通过 `config/index.js` 加载。

### 3. 数据库设计 (SQLite + FTS5)
针对 SQLite 默认不支持中文分词的痛点，我们将采用主流的**“预分词”策略**解决：
1. **`documents` (元数据表)**: 保存原始文档的绝对路径、相对路径、文件名、修改时间以及不包含格式的原始抽取文本。
2. **`documents_fts` (全文搜索表)**: 这是一个虚拟的 FTS5 引擎表。在文本入库前，后端代码会调用 `nodejieba` 将中文切分为词组并用空格隔开（例如：`测试文档引擎` -> `测试 文档 引擎`），并把分词后的字符串存入此表。
3. **查询机制**: 搜索时，先将用户的关键词进行同样的分词，之后对 FTS5 引擎执行 `MATCH` 查询。利用 FTS5 内置的高效率 `snippet` 能力，快速提取并返回含有关键词高亮片段的数据。

### 4. 数据同步与处理流程
- **目录扫描服务**: 服务器启动时全量扫描指定的文档目录（处理首次导入）。
- **文件监听引擎 (File Watcher)**: 可以使用 `chokidar` 库监听文件所在目录的变化，当本地目录有新增、修改、重命名或删除文件时，实时将其加入或移出 SQLite 数据库索引，无需人工干预。此模块完全独立运行，不阻塞主 API 进程。
- **构建索引耗时估算**: 200MB 数据的全量提取与分词预计时间在数分钟内；日常增量更新耗时通常在100毫秒内，对系统资源占用极小。

### 5. 前端界面设计 (重现 Google 风格)
- **首页 (`/`)**: 极简留白风格，包含页面中央的主 Logo、带圆角阴影的宽屏搜索框、以及下方的 "Google 搜索" 等按钮。
- **搜索结果页 (`/search?q=xxx`)**:
  - 顶部左侧固定小型 Logo 与搜索栏，方便直接修改关键词进行二次查找。
  - 中间大块区域展示搜索结果列表，呈现逻辑同 Google 搜索：
    - 文档标题：采用经典 Google 蓝字体 `(color: #1a0dab)`，**点击不跳转，而是弹出预览浮窗**。
    - 路径面包屑：采用绿色文本，提示文件所在目录层级。
    - 内容摘要 (Snippet)：展示文本片断，并将匹配到的关键词加粗显示或标红。
  - 底部配置简单的分页栏数字页码。
- **预览浮窗组件 (Modal)**:
  - 占据大半屏幕的中心浮层，带半透明遮罩背景。
  - 包含一个关闭按钮。
  - 主体大容器加载该文档的**纯文本完整内容**，保留换行符。提供内独立滚动条。
  - 底部区域提供该文档原始 **`.doc` 或 `.docx` 文件的直接下载链接/按钮**。

### 6. 核心接口规范 (对外开放 API)
系统采用 RESTful API 设计标准，不仅为自身前端提供服务，外部程序也可直接调用以下接口：

1. **`GET /api/search`** (文档检索接口)
   - 查询参数：`q` (搜索词), `page` (页码, 默认1), `limit` (每条页数, 默认10)
   - 响应格式：JSON，包含 `total` (总数), `page`, `limit`, 以及 `data` 数组 (每项包含 id、标题、路径、高亮片段 snippet、最后修改时间等)。
2. **`GET /api/document/:id/text`** (获取全文纯文本预览)
   - 查询特定文档的内容 (例如供前端浮窗使用)。
   - 响应格式：JSON，包含 `content` (无格式纯文本，保留换行), `title` 等。
3. **`GET /api/document/:id/download`** (直接下载原文件)
   - 直接返回原始 `.doc` 或 `.docx` 的文件二进制流，设置 `Content-Disposition: attachment`。
4. **`GET /api/system/status`** (系统运行状态监控)
   - 返回当前系统已索引文件总数、是否正在进行全量更新、服务启动时间等状态。

## 7. 实施与开发计划的重点

1. **统一配置与解耦结构**: 严格按照前述的三层架构（Controller -> Service -> Model）落地工程开发，并在第一步建立 `config` 单点配置。
2. **老式 `.doc` 文件解析**: 按原计划默认优先使用纯 Node 组件（例如 `word-extractor`）抽取纯文本以便入库与预览时使用。
3. **极简前端架构**: 采用**纯原生 HTML / CSS / Vanilla JS**，前端代码直接存放在静态目录，后端提供静态文件服务 (`express.static`)。前端通过 Fetch API 与后端交互，运用 DOM 操作实现无刷新搜索及弹窗交互。
